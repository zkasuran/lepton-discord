The loop

Decide → pay → answer.

The agent's reasoning is free. It only spends USDC on the external tools it decides are worth buying, and never past your budget.

01 · DECIDE

It reasons over price

Given your question and a priced tool catalog, the agent picks the single cheapest tool that helps, or answers for free. Over-budget spends are declined in code, not left to the model.

02 · PAY

It signs, never gasses

It signs an EIP-3009 authorization from its own wallet (USDC only, no gas). A facilitator settles transferWithAuthorization on Arc in under 500ms and pays the gas.

03 · ANSWER

It hands you a receipt

Real data comes back (CoinGecko, Open-Meteo, Google News), the agent composes the answer, and posts it with a clickable Arc receipt. Its own on-chain tx count stays 0.

For the judges

Every question, answered.

The honest version, up front.

Is the money real?

Yes. Real testnet USDC settles between two distinct wallets on Arc, status 1, and you can click every receipt on this page. The five settlements above are permanent on-chain records that need no server to verify. When the live box is running, each paid ask produces a fresh settlement of its own.

Is the agent really deciding, or is it scripted?

It is a real model tool-use call over a priced catalog. It can buy a tool, pick a different tool, or answer for free and spend nothing. Ask it a greeting and it spends $0; ask for a live price and it buys one. The budget is enforced in code after the model picks, so a hallucinated number can never overspend.

Why does the agent show 0 on-chain transactions if it paid?

EIP-3009, the x402 "exact" scheme. The agent signs a transfer authorization off-chain; a facilitator broadcasts it and pays the gas. On every receipt the from is the facilitator, not the agent, and the to is the USDC contract. The payer needs USDC, nothing else.

What is genuinely yours versus the Circle SDK?

The agent's decision and per-user budget layer, the real executors behind the paywall, the two-wallet architecture, the embedded facilitator wiring, and the Discord surface itself. The x402 protocol plumbing is the SDK. The agentic commerce on top of it is the build.

Which Circle tools does it use?

The x402 exact scheme, EIP-3009 USDC on Arc, an embedded facilitator that settles in-process, and USDC-as-gas. The two-wallet split (agent payer, service facilitator) is what makes each settlement a real transfer between distinct parties, not a wallet paying itself.

Real users yet?

Yes, in the event window. The bot runs 24/7, so this is genuine usage: distinct Discord users have paid in the channel, each a real USDC settlement on Arc. The figures come from the payment store via the traction report, not hand-typed, and real Discord users are counted separately from the public web demo. We report the real number, never an inflated one.
